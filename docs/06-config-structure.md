Dotter uses TOML configuration files to define packages, files, and variables. This page covers the structure and format of those files.

## Configuration Files Overview

Dotter uses two main configuration files:

| File | Purpose | Typical Location |
|------|---------|------------------|
| `global.toml` | Defines packages, files, variables, and helpers | `.dotter/global.toml` |
| `local.toml` | Selects which packages to enable, can override variables | `.dotter/local.toml` |

Additionally, Dotter uses:

| File | Purpose |
|------|---------|
| `.dotter/cache.toml` | Tracks what Dotter has deployed |
| `.dotter/cache/` | Stores rendered template copies for comparison |
| `.dotter/<hostname>.toml` | Alternative machine-specific local config |
| Hook scripts | Optional scripts that run before/after operations |

## global.toml

The global configuration defines your packages and shared settings.

### Package Definition

Packages are defined as top-level TOML tables:

```toml
[shell]
files = { "bashrc" = "~/.bashrc" }
variables = { shell = "bash" }
depends = []

[vim]
files = { "vimrc" = "~/.vimrc" }
depends = ["shell"]
```

### Package Sections

Each package can have:

#### files

Maps source files to target locations:

```toml
[my_package.files]
"source_file" = "~/.target_file"
"config/app" = "~/.config/app"
```

#### variables

Defines variables for templating:

```toml
[my_package.variables]
name = "value"
count = 42
nested = { key = "value" }
```

#### depends

Lists packages that this package depends on:

```toml
[vim]
depends = ["shell"]

[vim.files]
"vimrc" = "~/.vimrc"
```

When you enable `vim`, the `shell` package is automatically enabled too.

### Settings Section

The `[settings]` table controls global behavior:

```toml
[settings]
default_target_type = "symbolic"  # or "template" or "automatic"
```

#### default_target_type

Controls how files without explicit type are handled:

| Value | Behavior |
|-------|----------|
| `"automatic"` | Template if file contains `{{`, otherwise symlink (default) |
| `"symbolic"` | Files without explicit type become symlinks |
| `"template"` | Files without explicit type become templates |

Example:

```toml
[settings]
default_target_type = "template"

[default.files]
"bashrc" = "~/.bashrc"  # Will be templated even without {{ }}
```

**Note**: This setting is applied before `local.toml` patches are merged. Files added in `local.toml`'s `[files]` section will NOT be converted by this setting — they remain as `Automatic` type.

### Helpers Section

Define custom Handlebars helper scripts:

```toml
[helpers]
my_helper = "scripts/my_helper.sh"
```

The helper can then be used in templates:

```handlebars
{{my_helper "argument"}}
```

Helpers require the `scripting` feature (enabled by default). See the [Hooks](hooks) page for details on how helpers work.

## File Target Types

There are three ways to specify file targets, giving you control over how files are deployed.

### Automatic (Simple Form)

The simplest form — just the target path:

```toml
[default.files]
"bashrc" = "~/.bashrc"
```

Dotter automatically determines symlink vs. template based on file contents.

### Symbolic Link

Explicitly create a symbolic link:

```toml
[default.files]
"bashrc" = { target = "~/.bashrc", type = "symbolic" }
```

With additional options:

```toml
[default.files]
"bashrc" = { target = "~/.bashrc", type = "symbolic", owner = "root", recurse = false, if = "is_linux" }
```

#### Symbolic Link Options

| Option | Type | Description |
|--------|------|-------------|
| `target` | string | Target path (required) |
| `type` | string | Must be `"symbolic"` |
| `owner` | string or number | User who should own the file (Unix only) |
| `recurse` | boolean | Whether to recurse directories |
| `if` | string | Condition expression |

### Template

Explicitly treat as a template:

```toml
[default.files]
"bashrc" = { target = "~/.bashrc", type = "template" }
```

With additional options:

```toml
[default.files]
"bashrc" = { target = "~/.bashrc", type = "template", prepend = "# Generated by Dotter\n", append = "\n# End", if = "is_linux" }
```

#### Template Options

| Option | Type | Description |
|--------|------|-------------|
| `target` | string | Target path (required) |
| `type` | string | Must be `"template"` |
| `owner` | string or number | User who should own the file (Unix only) |
| `prepend` | string | Text to add at the beginning |
| `append` | string | Text to add at the end |
| `if` | string | Condition expression |

### Conditional Files with `if`

The `if` field lets you conditionally include files based on your variables:

```toml
[default.files]
"linux_bashrc" = { target = "~/.bashrc", if = "dotter.linux" }
"macos_bashrc" = { target = "~/.bashrc", if = "dotter.macos" }
```

The condition is evaluated as a Handlebars expression. If it evaluates to truthy, the file is deployed.

**Syntax note**: When referencing a variable directly (like `dotter.linux`), no parentheses are needed. When calling a helper function (like `is_executable`), wrap it in parentheses as a subexpression:

```toml
# Variable reference — no parentheses
"linux_file" = { target = "~/.linux", if = "dotter.linux" }

# Helper call — requires parentheses
"vimrc" = { target = "~/.vimrc", if = "(is_executable \"vim\")" }
```

In TOML, quotes inside the string must be escaped with backslashes.

See the [Built-In Variables](templates#built-in-variables) section for what you can test.

### Directory Handling

If the source is a directory, Dotter recursively expands it into individual file mappings:

```toml
[default.files]
"config" = "~/.config"
```

This deploys all files inside `config/` to `~/.config/`.

The `recurse` option (for symbolic links) controls this:

```toml
[default.files]
"config" = { target = "~/.config", type = "symbolic", recurse = false }
```

With `recurse = false`, the entire directory is symlinked as a single unit.

### Owner Specification (Unix Only)

On Unix systems, you can specify file ownership:

```toml
[default.files]
"root_config" = { target = "/etc/myapp.conf", type = "template", owner = "root" }

# Or by UID
"system_config" = { target = "/etc/system.conf", type = "template", owner = 0 }
```

When `owner` is specified, Dotter uses `sudo` to create files as that user.

## local.toml

The local configuration selects packages and can override variables.

### Package Selection

```toml
packages = ["shell", "vim", "git"]
```

Only listed packages (and their dependencies) are deployed.

### Variable Overrides

```toml
packages = ["default"]

[variables]
username = "myuser"
font_size = 14
```

Variables here override or add to those defined in `global.toml`.

### File Overrides

Add files not in the global config:

```toml
packages = ["default"]

[files]
"extra_config" = "~/.extra"
```

Or override file targets:

```toml
[files]
"bashrc" = ""  # Remove this file from deployment
```

Setting a target to an empty string excludes the file.

### Includes

Load additional configuration from other files:

```toml
packages = ["default"]
includes = ["extra.toml"]
```

Files are loaded in order, and later files override earlier ones.

## Hostname-Based Config

Instead of `local.toml`, you can name your local config after your machine's hostname:

```bash
# If your hostname is "laptop"
.dotter/laptop.toml
```

Dotter automatically uses `<hostname>.toml` if `local.toml` does not exist.

This is useful for having machine-specific configurations in the same repository:

```text
.dotter/
├── global.toml
├── laptop.toml
├── desktop.toml
└── work.toml
```

Each machine uses its own config based on hostname.

## How Configuration Is Loaded

Understanding the order in which Dotter loads and merges configuration is essential for debugging unexpected behavior.

### Loading Order

1. **global.toml** is loaded first. This defines all packages, their files, variables, helpers, and settings.

2. **includes** are processed. If `local.toml` has an `includes` list, each included file is loaded in order. For each included file:
   - If a package exists in both global and included, the included file's files and variables are merged into the global package
   - If a package only exists in the included file, it is added to the global config
   - Files with the same source path within a package override the global definition
   - Variables with the same name are merged recursively (nested tables merge, scalars are overwritten)

3. **Package filtering**. Only packages listed in `local.toml`'s `packages` array (and their transitive dependencies) are kept. All other packages are discarded.

4. **Package merging**. All remaining packages are merged into a single configuration:
   - Files from each package are combined. If the same source path appears in multiple packages, this is an error.
   - Variables from each package are merged. If the same variable name appears in multiple packages, nested tables are merged recursively, but scalar values are an error (duplicate variable).

5. **default_target_type** is applied. Any file without an explicit `type` is converted according to the setting. **Note**: This happens before local.toml patches are applied (step 6), so files added in `local.toml`'s `[files]` section will NOT be converted by this setting — they remain as `Automatic` type.

6. **local.toml** patches are applied. The `[files]` and `[variables]` sections in `local.toml` are merged on top:
   - Files with the same source path override the global definition
   - Variables with the same name are merged recursively (nested tables merge, scalars are overwritten)

7. **stdin patch** (if `--patch` is used). Additional files and variables from stdin are applied on top of everything else.

8. **Empty target removal**. Files with an empty string target (`""`) are removed.

9. **Directory expansion**. If any source is a directory, it is recursively expanded into individual file mappings.

10. **Tilde expansion**. All `~` in file paths are expanded to the home directory.

11. **Condition filtering**. Files with an `if` condition are evaluated. If the condition is false, the file is removed from the deployment.

### Variable Merging Rules

Variables merge recursively. This means nested tables are combined, not replaced:

```toml
# global.toml
[default.variables.theme]
background = "#282a36"
foreground = "#f8f8f2"

[default.variables.font]
name = "FiraCode"
size = 12
```

```toml
# local.toml
packages = ["default"]

[variables.theme]
background = "#000000"  # Overrides only this key

[variables.font]
size = 14  # Overrides only this key
```

Result:

```toml
# Final merged variables
[variables.theme]
background = "#000000"   # Overridden by local.toml
foreground = "#f8f8f2"   # Kept from global.toml

[variables.font]
name = "FiraCode"  # Kept from global.toml
size = 14          # Overridden by local.toml
```

**Important**: If the same scalar variable name appears in two different packages (not through includes or local overrides), this is an error. Dotter will refuse to merge them.

### Includes Behavior

The `includes` field in `local.toml` lets you split configuration across multiple files:

```toml
# local.toml
packages = ["default"]
includes = ["shared.toml", "secrets.toml"]
```

Each included file is processed in order:

1. The included file's packages are merged into the global packages
2. Files with the same source path within a package override the global definition
3. Variables with the same name are merged recursively

This is useful for:

- Sharing common settings across machines
- Keeping secrets in a separate, uncommitted file
- Organizing large configurations

### Package Dependencies

Dependencies are resolved transitively. If package A depends on B, and B depends on C, enabling A enables all three:

```toml
[a]
depends = ["b"]

[b]
depends = ["c"]

[c]
files = { "config" = "~/.config" }
```

```toml
# local.toml
packages = ["a"]  # Also enables "b" and "c"
```

Dotter iterates until no new packages are added, so circular dependencies are handled gracefully (they just enable each other).

## Automatic Type Detection

When you use the simple form (`"file" = "~/.target"`) or set `default_target_type = "automatic"`, Dotter decides whether to symlink or template each file.

### Detection Algorithm

For each file:

1. If the source is **not valid UTF-8** (binary file), it is treated as a **symlink**
2. If the source is a **directory**, it is treated as a **symlink** (or expanded recursively)
3. If the source file **contains `{{`** anywhere in its content, it is treated as a **template**
4. Otherwise, it is treated as a **symlink**

### What This Means in Practice

- A file with `{{username}}` is always templated
- A file with `{{` in a comment is also templated (even if you did not intend it)
- Binary files like images or compiled executables are always symlinked
- If you want a file with `{{` to be symlinked, use explicit type:

```toml
[default.files]
"readme" = { target = "~/.readme", type = "symbolic" }
```

### Windows Fallback

On Windows without Developer Mode, all files are treated as templates regardless of detection. This is because symlinks require elevated privileges.

## Cache Structure

Dotter maintains a cache to track what it has deployed. This is how `undeploy` knows what to remove.

### cache.toml

```toml
[symlinks]
"bashrc" = "~/.bashrc"
"vimrc" = "~/.vimrc"

[templates]
"gitconfig" = "~/.gitconfig"
```

- **symlinks**: Maps source paths (in your repository) to target paths (deployment locations) for symlinks
- **templates**: Maps source paths to target paths for templates

The cache is updated after every deploy and undeploy operation.

### cache/ Directory

The cache directory stores rendered copies of template files. This allows Dotter to compare the current deployed file against what it last rendered.

```text
.dotter/
├── cache.toml
└── cache/
    └── gitconfig       # Rendered copy of the template
                        # (symlinks have no cache/ entry)
```

When you run `dotter`:

1. Dotter renders the template
2. Compares the rendered output against the cache copy
3. If different, it copies the new rendered output to the target
4. Updates the cache copy

This means Dotter only writes to the target when the rendered content actually changed.

### Smart Re-deploy

If you manually edit a deployed template file and the new rendered output from the source matches your manual edit, Dotter will re-deploy it without error. This means if you happen to edit a deployed file to exactly match what Dotter would render, the next deploy succeeds normally.

### Cache and Undeploy

The cache is essential for `undeploy`. Without it, Dotter does not know what files to remove:

```bash
# This will fail if cache is missing
dotter undeploy
# Error: load cache: Cannot undeploy without a cache.
```

If you lose your cache, you must manually remove deployed files.

### Cache and Repository Moves

If you move your dotfiles repository, the cache still contains absolute paths to the old source locations. Symlinks will break. To fix:

1. Run `dotter undeploy` from the old location (if possible)
2. Or manually remove the symlinks and cache
3. Run `dotter` from the new location

## Example Configuration Files

### Minimal global.toml

```toml
[default.files]
"bashrc" = "~/.bashrc"
"vimrc" = "~/.vimrc"

[default.variables]
editor = "vim"
```

### Minimal local.toml

```toml
packages = ["default"]
```

### Advanced global.toml

```toml
[settings]
default_target_type = "automatic"

[helpers]
custom = "scripts/helper.sh"

# Base shell configuration
[shell]
depends = []

[shell.files]
"bashrc" = "~/.bashrc"
"profile" = "~/.profile"

[shell.variables]
prompt = "> "

# Vim with dependency on shell
[vim]
depends = ["shell"]

[vim.files]
"vimrc" = "~/.vimrc"
"vim" = { target = "~/.vim", type = "symbolic", recurse = false }

[vim.variables]
leader = ","

# Git configuration
[git.files]
"gitconfig" = { target = "~/.gitconfig", type = "template" }
"gitignore_global" = "~/.gitignore"

[git.variables]
git_name = "Your Name"
git_email = "your.email@example.com"

# Conditional Linux-only config
[linux_only.files]
"linux_config" = { target = "~/.config/linux", if = "dotter.linux" }
```

### Advanced local.toml

```toml
packages = ["shell", "vim", "git"]

[variables]
# Override vim leader for this machine
leader = ";"

# Custom prompt
prompt = "$ "

# Add extra file for this machine
[files]
"local_script" = "~/.local/bin/script"

# Disable gitconfig (use system default)
"gitconfig" = ""
```
